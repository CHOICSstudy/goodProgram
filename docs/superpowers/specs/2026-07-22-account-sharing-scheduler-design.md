# 인강 계정 공유 시간 관리 웹페이지 — 설계 문서

**작성일**: 2026-07-22
**프로젝트명**: goodProgram

## 배경 & 문제

팀이 인터넷 강의(인강)를 여러 계정으로 나눠 결제해 공유한다. 인강 사이트는 한 계정에 동시 접속이 안 되는데, "지금 누가 어떤 계정에 접속 중인지" 관리가 안 돼서 동시 접속 충돌이 자주 발생한다. 이를 해결하는 시간 관리 웹페이지를 만든다.

- 사용 인원: 팀원 5명 내부용 (서로 신뢰 기반)
- 목표: 계정 점유 현황을 공유하고, 미리 예약해 충돌을 방지

## 핵심 기능

1. 인강 목록 — 카테고리(backend/frontend/server 등) 필터 + 제목 검색
2. 각 인강이 묶인 계정 표시 + 현재 사용 가능 여부(사용 가능 / 사용 중)
3. "계정 확인하기" — 로그인해야 할 사이트/계정 정보 + 아이디·비밀번호 복사
4. 시간표 예약 — 계정별 주간 예약표(30분 슬롯), 충돌 방지
5. 점유 판단 — 예약 + 실제 체크인/체크아웃 병행
6. 진입 보안 — 팀 공용 비밀번호 게이트 → 이름 선택

## 기술 스택

- **Next.js (App Router)** — 프론트 + 서버(Server Actions / Route Handlers), Vercel 배포
- **Supabase (Postgres)** — 데이터 저장소, 서버 측에서만 접근
- **상태 갱신** — 몇 초 간격 폴링(5명 규모에 충분, 필요 시 실시간 구독으로 업그레이드 가능)

## 아키텍처

- Next.js 하나로 프론트와 서버 로직을 모두 처리. 민감 정보(계정 비밀번호)는 서버에서만 다루고 클라이언트로 직접 노출하지 않는다.
- Supabase Postgres 접근은 서버 코드에서만. 접속 키와 공용 비밀번호는 환경 변수로 관리(하드코딩 금지).
- "사용 중" 상태·예약 현황은 클라이언트가 몇 초 간격으로 폴링해 갱신.

## 데이터 모델

```
accounts (인강 사이트 계정)
  id, label(예: "계정1"), site(어느 인강 사이트),
  login_id, login_password, created_at

courses (인강)
  id, title, category(backend|frontend|server|...),
  account_id → accounts.id

reservations (예약)
  id, course_id → courses.id, account_id → accounts.id,
  member_name, start_at, end_at, created_at
  제약: 같은 account_id + 겹치는 시간대 중복 예약 불가

sessions (실제 체크인 = 지금 점유)
  id, account_id → accounts.id, member_name,
  checked_in_at,
  planned_checkout_at(null = 미정),
  checked_out_at(null = 아직 안 나감 / 자동 해제 전)

login_attempts (공용 비밀번호 무차별 시도 방지)
  ip, fail_count, locked_until, updated_at
```

## 핵심 로직

### "현재 사용 가능 여부" 판단 (읽을 때 계산)

계정에 대해 **활성 세션**이 있으면 "사용 중", 없으면 "사용 가능".

활성 세션 조건:
- `checked_out_at`이 null 이고,
- 아직 **예정 해제 시각 전**이다.
  - 예정 해제 시각 = `planned_checkout_at`이 있으면 그 값, `null`(미정)이면 `checked_in_at + 2시간`.

즉 별도 스케줄러 없이 읽는 시점에 계산한다. 미정 세션은 체크인 2시간 후 자동으로 "사용 가능"이 된다.

### 체크인 흐름

- 체크인 시 **체크아웃 예정 시간을 지정**:
  - 특정 시간 지정 → 그 시각에 자동 해제
  - "미정" 선택 → 체크인 2시간 후 자동 해제
- 체크인/체크아웃은 "계정 확인하기" 모달 안에서도, 모달을 열지 않고 인강 카드에서 바로도 가능.

### 예약

- 계정별 주간 예약표. 열: 월~일, 행: 09:00~24:00 30분 단위.
- 빈 슬롯 클릭 → 예약(내 이름 자동). 예약된 슬롯은 잠김 + 초록색 + 예약자 이름.
- 같은 계정·겹치는 시간대 중복 예약은 DB 제약으로 방지.
- 인강 카드에는 **당일 예약만** 참고 텍스트로 표시("다음 예약: 홍길동 14:00~15:00"). 내일 이후 예약은 시간표에서만 확인.

## 화면 구성

### 진입 흐름
1. 팀 공용 비밀번호 입력 (서버 검증 → 세션 쿠키)
2. 이름 선택 (5명)
3. 메인 페이지 이용

### 페이지 1: 인강 목록
- 인강 카드 리스트 + 상단 카테고리 필터 + 제목 검색창
- 카드 표시: 제목, 카테고리, 묶인 계정 라벨, 현재 상태 배지(🟢 사용 가능 / 🔴 사용 중 + 사용자 이름), 당일 예약 참고 텍스트
- 카드에서 바로 체크인/체크아웃 가능
- "계정 확인하기" 버튼 → 모달:
  - 로그인할 사이트 / 계정 정보
  - 아이디·비밀번호 복사 버튼 (비밀번호는 기본 가림 + "보기" 토글)
  - 체크인/체크아웃 버튼 (체크아웃 예정 시간 지정)

### 페이지 2: 시간표(예약)
- 왼쪽 사이드바에 계정 리스트 세로 나열
- 계정 클릭 → 오른쪽에 그 계정의 주간 예약표(월~일 × 09:00~24:00 30분)
- 예약 슬롯 잠김/초록/예약자 이름, 빈 슬롯 흰색
- 내 예약은 취소 가능

### 관리 화면 (간단)
- 인강 추가/수정/삭제, 계정 추가/수정/삭제(로그인 정보 포함). 5명 신뢰 기반이라 별도 관리자 권한 없음.

## 보안

- 진입 시 팀 공용 비밀번호 게이트(서버 검증 → 통과 시 세션 쿠키).
- **무차별 시도 방지**: 5번 연속 틀리면 1분간 입력 차단("1분 후 다시 시도"). 성공 시 실패 카운트 초기화. Vercel 서버리스라 상태를 메모리에 못 담으므로 `login_attempts` 테이블에 IP별로 기록(요청 시 `locked_until` 확인, 5회 도달 시 `locked_until = 지금 + 1분`).
- 계정 로그인 정보는 서버에서만 조회, 게이트 통과한 사용자에게만 전달.
- 공용 비밀번호·Supabase 키는 환경 변수로 관리.

> **주의**: 이름 선택만으로는 개인 인증이 안 되므로, 실질적 보호는 공용 비밀번호 게이트에 의존한다. URL + 공용 비밀번호를 아는 사람은 모든 계정 정보를 볼 수 있음(내부 5명 신뢰 모델로 수용).

## 에러 처리 (필요한 것만)

- 예약 충돌: 같은 계정·같은 슬롯 동시 예약 → DB 제약으로 막고 "이미 예약됨" 안내
- 체크인 충돌: 이미 사용 중인 계정에 체크인 → "OO님이 사용 중" 안내
- 공용 비밀번호 오류 → 재입력 안내 (+ 위 무차별 차단)

## 테스트

- 핵심 로직 단위 테스트: "사용 가능 여부" 판단(자동 체크아웃 포함), 예약 충돌 방지, 로그인 무차별 차단.
- UI는 최소한의 수동 확인.

## 범위 밖 (YAGNI)

- 실시간 구독(초기엔 폴링), 개인 인증/권한, 모바일 앱, 알림, 인강 외 다른 사이트 연동.
